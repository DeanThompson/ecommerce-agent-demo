## User Scenarios & Testing *(mandatory)*

### User Story 1 - 订单数据查询与分析 (Priority: P1)

作为物流负责人，我想通过自然语言查询订单配送情况，以便了解物流效率并优化配送策略。

**Why this priority**: 订单是电商核心业务，配送时效直接影响客户满意度。订单查询是继销售查询后最重要的分析维度。

**Independent Test**: 可通过输入"订单状态分布"或"平均配送时间"等问题独立测试，验证系统能返回正确的订单统计数据和可视化图表。

**Acceptance Scenarios**:

1. **Given** 用户已进入对话界面，**When** 用户输入"2017年订单状态分布"，**Then** 系统返回各状态订单数量的饼图和文字说明
2. **Given** 用户已进入对话界面，**When** 用户输入"平均配送时间是多少天"，**Then** 系统返回平均配送天数的数字卡片
3. **Given** 用户已进入对话界面，**When** 用户输入"各州的平均配送时间对比"，**Then** 系统返回按州分组的配送时间柱状图
4. **Given** 用户已进入对话界面，**When** 用户输入"配送超时的订单有多少"，**Then** 系统返回超时订单数量、占比及相关分析

---

### User Story 2 - 时间趋势分析 (Priority: P1)

作为运营人员，我想查看销售和订单的时间趋势，以便识别业务规律和制定运营策略。

**Why this priority**: 趋势分析是数据分析的核心能力，帮助用户理解业务变化规律，与订单查询同等重要。

**Independent Test**: 可通过输入"按月展示2017年销售趋势"独立测试，验证系统能生成正确的折线图展示时间序列数据。

**Acceptance Scenarios**:

1. **Given** 用户已进入对话界面，**When** 用户输入"按月展示2017年销售趋势"，**Then** 系统返回12个月销售额的折线图
2. **Given** 用户已进入对话界面，**When** 用户输入"2017年和2016年销售对比"，**Then** 系统返回两年销售额的对比折线图或柱状图
3. **Given** 用户已进入对话界面，**When** 用户输入"订单量的周趋势"，**Then** 系统返回按周汇总的订单量折线图
4. **Given** 用户已进入对话界面，**When** 用户输入"Q4季度销售环比增长"，**Then** 系统返回环比增长数据和趋势分析

2. **Given** 用户已进入对话界面，**When** 用户输入"有多少客户进行了复购"，**Then** 系统返回复购客户数量、占比的数字卡片
3. **Given** 用户已进入对话界面，**When** 用户输入"客单价分布情况"，**Then** 系统返回客单价分布的柱状图

### User Story 5 - 卖家分析 (Priority: P3)

### Edge Cases

- 当查询的时间范围内没有数据时，系统应返回友好提示"该时间范围内暂无数据"
- 当用户查询的维度组合过于复杂导致结果集过大时，系统应自动限制返回数量并提示用户
- 当用户输入的查询意图模糊时，系统应请求用户澄清或提供可能的查询选项
- 当数据库查询超时时，系统应返回超时提示并建议用户缩小查询范围
- 当图表数据点过多影响可读性时，系统应自动聚合或提供分页展示

## Requirements *(mandatory)*

### Functional Requirements

**MCP 工具要求**

- **FR-001**: 系统必须提供 `query_orders` 工具，支持按时间范围、订单状态、地区筛选订单数据
- **FR-002**: 系统必须提供 `query_customers` 工具，支持客户分布分析、复购分析、客单价分析
- **FR-003**: 系统必须提供 `query_reviews` 工具，支持评分筛选、按品类/时间分组统计
- **FR-004**: 系统必须提供 `query_sellers` 工具，支持卖家排名、地域分布、绩效查询
- **FR-005**: 系统必须提供 `analyze_trend` 工具，支持按日/周/月/季度的时间序列分析，支持同比、环比计算

**图表可视化要求**

- **FR-006**: 系统必须支持折线图 (line) 类型，用于展示时间序列趋势数据
- **FR-007**: 系统必须支持柱状图 (bar) 类型，用于展示分类对比数据
- **FR-008**: 系统必须支持饼图 (pie) 类型，用于展示占比分布数据
- **FR-009**: 折线图必须支持多系列展示，用于同比/环比对比
- **FR-010**: 柱状图必须支持横向和纵向两种展示方式
- **FR-011**: 所有图表必须支持悬停显示具体数值

**数据查询要求**

- **FR-012**: 订单查询必须支持计算配送时效（实际送达时间 - 下单时间）
- **FR-013**: 订单查询必须支持识别配送超时订单（实际送达时间 > 预计送达时间）
- **FR-014**: 客户查询必须支持基于 customer_unique_id 识别复购客户
- **FR-015**: 评价查询必须支持按评分范围筛选（如差评定义为评分 <= 2）
- **FR-016**: 趋势分析必须支持计算同比增长率和环比增长率

**交互要求**

- **FR-017**: Agent 必须能根据数据特征自动选择合适的图表类型
- **FR-018**: 系统必须在图表旁提供文字洞察和分析说明

### Key Entities

- **Order (订单)**: 包含订单ID、客户ID、订单状态、下单时间、发货时间、送达时间、预计送达时间。关联客户、订单明细、支付、评价。
- **Customer (客户)**: 包含客户ID、唯一标识、所在城市、所在州。用于客户分布和复购分析。
- **Review (评价)**: 包含评价ID、订单ID、评分(1-5)、评价内容、创建时间。用于满意度分析。
- **Seller (卖家)**: 包含卖家ID、所在城市、所在州。关联订单明细，用于卖家绩效分析。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在单次对话中完成订单、客户、评价、卖家任一维度的数据查询
- **SC-002**: 时间趋势查询能在 3 秒内返回结果并展示折线图
- **SC-003**: 系统能正确识别并展示同比/环比增长率，误差不超过 0.1%
- **SC-004**: 90% 的趋势类查询自动使用折线图展示，90% 的占比类查询自动使用饼图展示
- **SC-005**: 用户能通过自然语言完成至少 10 种不同类型的数据分析查询
- **SC-006**: 图表悬停交互响应时间小于 100 毫秒

## Assumptions

- Spec 1 (MVP 框架) 已完成，包括基础对话功能、SSE 流式响应、`query_sales` 和 `generate_chart` 工具
- 数据库已导入完整的 Olist 数据集（9 张表 + 2 个视图）
- 前端已有基础的对话组件和 metric/table 图表组件
- 差评定义为评分 <= 2 分
- 配送超时定义为实际送达时间晚于预计送达时间
- 复购客户定义为 customer_unique_id 出现在多个订单中的客户
